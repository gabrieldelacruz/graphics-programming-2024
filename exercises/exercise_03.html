Exercise 03 - Transformations: Placing gears

Learning objectives

This week we will introduce transformation matrices that allow us to place objects on the scene. Specifically, we are going to:

  • Manipulate vectors and transformation matrices using GLM.
  • Practice translation, rotation and scale, and understand the order in which they are applied.
  • Learn how to link objects using their transformations matrices.
  • Start using the camera to render 3D scenes.
  • Visualize the difference between object space, world space and clip space (NDC).

Remember that you can find hints in comments in the code, tagged with the label "(todo)" followed by the number of the exercise.

References

  • LearnOpenGL: Coordinate systems: https://learnopengl.com/Getting-started/Coordinate-Systems

Exercise 03.1 - Rotating the gear

Your first task is to make the gear rotate. To do this, instead of modifying the vertices in the VBO, we will transform them in the shader.

Add a new uniform to the vertex shader, a matrix (mat4) named WorldMatrix. You can give it an initial value to be used until we set the value. A good value would be the identity matrix, that you can create like this: mat4(1.0f)

Use it to multiply the value of the vertex position and assign it to gl_Position. Because matrix multiplications are right-to-left in OpenGL, you should place the matrix at the left of the position vector.

To set the uniform value from the C++ code, we need to find the uniform location in the shader. Create a member variable of type ShaderProgram::Location and store the value of the uniform location, during initialization. You can query the location by name using the shader program.

The actual value of the matrix has to be set every frame during rendering. We will do so inside the helper function DrawGear.

Now that everything is set up, we need to generate the rotation matrix. GLM library has some helper functions that we will use. For rotation, you can use a function that takes an angle and the rotation axis: glm::rotate. For the angle, we need a global rotation speed in radians (start with a value of 1, for instance), multiplied by the current time. Can you figure out the rotation axis?

Exercise 03.2 - Moving gears and matching rotation speed

We will render a new gear, next to the current one. Add a new DrawGear in the render function. For the mesh, use the medium gear. You can specify any color you want, for this one and for the large gear. Start with the identity matrix (glm::mat(1.0)) and run the program to make sure the new gear is there.

Move the gear by using a translation matrix. You can create one using the GLM function glm::translate. The new position, to the right of the large one, will be (0.75f, 0, 0). Run the application and check the two gears next to each other.

The next step is to make the gear rotate in its new position. Create a rotation matrix and multiply it with the translation matrix. The order is important, do you have to multiply it from the left or the right? Try both and observe the results.

If you tried with the same speed, you may have noticed that the cogs don't fit. Can you adjust the speed of the new gear?
Hint: The rotation speed is INVERSELY PROPORTIONAL to the number of cogs (and in the opposite direction).

Exercise 03.3 - Finding the scale

Add a third gear, with the small gear mesh, centered at the top-left corner of the window (-1, 1, 0).

As it is now it is too small to connect with the other ones. Multiply the transform matrix by a scale matrix with glm::scale. From which side do we multiply the scale and translation matrices? Try both and observe the difference. Also, try to manually find a good value that connects the 2 gears.

Finally, make it rotate with a matching speed. Again... where does the rotation fit with the scale and the translation? Try different combinations until you get it right.

You may need an offset this time, before you adjust the speed. Set the global speed to 0 and add or substract a constant to the rotation until it matches with the cogs of the large gear. Then compute the relative speed for this gear and restore the value of the global speed.

Did you get the 3 gears moving in sync? Congratulations if you did!.

Sometimes we want some objects to be linked to the translation or rotations of others. To achieve this, we multiply the transformation matrices of those objects.

Render a new gear, the small again, at a position of (0, 0.2, 0).

Exercise 03.5 - Moving the camera

Before adding the camera, let's try something. What happens if we resize the window on one side, ending up with a window that is not square?

This happens because, without a view or projection matrix, the world space matches the clip space, in normal device coordinates, between -1 and 1. So, if we stretch the window, we stretch the world to fit between -1 and 1.

To fix this, we can add a camera to the application. Follow these steps:

  1. Include the camera header file in our application header.
  2. Create a member variable of type Camera.
  3. Open Camera.cpp and implement the missing methods (one line each).
  4. Add a uniform mat4 for the view-projection matrix (ViewProjMatrix).
  5. In the vertex shader, multiply the ViewProjMatrix with the WorldMatrix and the VertexPosition.
  6. Repeat the steps we did for the world matrix uniform, creating a member variable for the location and initializing it.
  7. In the Render() function, set the value of the new uniform to the camera view-projection matrix.

By default, both view and projection matrix contain the identity matrix. We want to set the value of the projection matrix to an orthographic projection, and use the view matrix to move the camera. Because we want it to change dynamically, we will set the values during the Update().

The orthographic projection method receives the min and max bounds of the volume that it will render. To see the same area as before, we will use a volume from -1 to 1. However, because we want it to adapt nicely to the window size, we will use the aspect ratio for the X component (-aspect to aspect). You can compute the aspect ratio by dividing the width of the window by the height (make sure that you avoid integer division).

After this, try resizing the window. Does it adapt now?

For the view matrix, we need to provide two parameters, the camera position, and the target position where we are looking at. One option is to leave the camera fixed at (0,0,1) and move the target following the mouse position, with coordinate Z set to 0. Another option, is to leave the target fixed at (0,0,0) and move the camera position with the mouse, with the Z coordinate set to 1. One will look around, the other one will orbit around the center, up to you!

If the mouse position does not work properly, make sure that you are requesting it normalized (by default it returns the position in pixels).

Once you modify the view matrix, you will notice that some parts of the scene disappear. This is because they get cut by the front plane, that is set at the same depth as the camera. Add some extra volume to the Z coordinates of the orthographic projection and it will be fixed.

Exercise 03.6 - Going 3D

Most of the exercises we have been doing until now look 2D, despite being rendered in a 3D world.

There are 2 things that contribute greatly to create a 3D look. One is perspective projection, which we will add shortly. The other one is lighting, which will have to wait until lecture 5.

We have everything we need for perspective projection, we just need to replace the orthographic projection we just added with a perspective projection matrix during update. For the parameters, we need the field of view (FOV), the aspect ratio, and the near and far planes. We already have the aspect ratio, and for near and far plane we can set any, as long as they contain the objects. You can try 0.1 and 10, for example.

As for the FOV, a value of PI/2 would be the closest to what we had. This is quite wide and will have some deformation, because the camera is really close to the objects. You can try different values and adjust the camera Z position accordingly.

Until we know more about lighting, we can try to fake something:

  1. Create an out variable in the vertex shader for the normals.
  2. Create the matching in variable in the fragment shader.
  3. Transform the normal to world coordinates and pass it to the fragment. Because normal is a vector, not a position, the 4th component of the vector should be 0 instead of 1.
  4. In the fragment shader, Multiply the color with this value: dot(normalize(Normal), vec3(1)) .

Does it change your perception of the scene?

Exercise 03.7 - Visualizing coordinate systems

Before you go to rest, a quick test to visualize different spaces.

Object space position: Add a new out Position variable and assign the value of the VertexPosition. In the fragment shader, replace the color with the new Position variable. How do the colors look like? Do they rotate with the object?

World space position: Using the same Position variable, assign the value of the VertexPosition transformed by the WorldMatrix. What are the main differences?

Clip space position: Once more, but this time assign the value of the VertexPosition transformed by the WorldMatrix AND the ViewProjMatrix. How does it look now? What happens if you move the camera?

If you want, you can repeat the experiment with the VertexNormal.
